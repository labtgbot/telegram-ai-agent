Спасибо, что помогаете делать Telegram AI Agent безопаснее. Эта политика
объясняет, как сообщить об уязвимости и чего ожидать в ответ.

  Top-10 чек-лист.

  ручного пентеста.
- [`docs/security/audit-report.md`](docs/security/audit-report.md) — отчёт

## Поддерживаемые версии

| Версия | Поддержка |
|--------|-----------|
| `main` (rolling) | ✅ Security fixes принимаются |
| Последний `v1.x` релиз | ✅ Critical / High fixes до следующего minor |
| Любая ветка старше 12 месяцев | ❌ Не поддерживается |

## Где сообщить об уязвимости

**Никогда** не открывайте публичный GitHub issue для уязвимости — это
поставит других пользователей под угрозу.

Используйте один из приватных каналов:

1. **GitHub Private Vulnerability Reporting (предпочтительный путь):**
   https://github.com/labtgbot/telegram-ai-agent/security/advisories/new —
   создайте draft advisory, мы получим уведомление автоматически.
2. **Email:** `security@labtgbot.example` (PGP-ключ запросите в первом
   письме — мы пришлём его в течение одного рабочего дня).
3. **Telegram:** для повторного контакта с уже знакомой security-командой
   можно использовать приватный канал, указанный в подписанном Engagement
   Letter.

## Что включить в отчёт

Чтобы мы могли быстро воспроизвести и закрыть проблему, приложите:

1. Краткое описание уязвимости и затронутого компонента (например,
   «IDOR в `GET /api/v1/admin/users/{id}`»).
2. Шаги воспроизведения — желательно с минимальным curl/Python-скриптом.
3. Влияние (impact) — какой класс данных или функциональности скомпрометирован.
4. CVSS-оценку, если вы её посчитали (мы пересчитаем независимо).
5. Идею или предложение по исправлению — опционально, но приветствуется.
6. Желаемые credits для будущего advisory (имя / handle / ничего).

## Что ожидать от нас

| Этап | SLA |
|------|-----|
| Подтверждение получения отчёта | 2 рабочих дня |
| Первичная триаж-оценка с приоритетом | 5 рабочих дней |
| Status-апдейт во время фикса | каждые 7 дней |
| Фикс P0/P1 в production | до 30 дней с момента подтверждения |
| Фикс P2 | до 90 дней |
| Публикация advisory + credits | в день релиза с патчем (или после согласованного embargo) |

При длительном расследовании мы предоставим точку контакта и регулярные
апдейты. Если SLA выше нарушается без объяснения, эскалируйте через тот же
канал — это будет видно maintainer-ам.

## Шкала серьёзности

См. подробности в [`docs/security/audit-report.md`](docs/security/audit-report.md).
Кратко:

- **P0 (Critical):** удалённое выполнение кода, компрометация прод-БД,
  доступ к деньгам / админу без авторизации. Public disclosure блокируется
  до выпуска фикса.
- **P1 (High):** компрометация одного аккаунта, persistent XSS, sustained
  DoS на public endpoint.
- **P2 (Medium):** ratelimit bypass, утечка неконфиденциальных метаданных,
  ошибочная конфигурация без сразу-выгоды для атакующего.
- **P3 (Low / Informational):** hardening, defence-in-depth предложения.

## Координированное раскрытие

Мы следуем модели координированного раскрытия (Coordinated Vulnerability
Disclosure):

1. Получаем отчёт → подтверждаем → согласуем embargo-дату (по умолчанию —
   90 дней или дата релиза с фиксом, что наступит раньше).
2. Готовим патч, выкатываем на production через стандартный pipeline
   (`release.yml` → Helm upgrade с reviewer approval).
3. Публикуем GitHub Security Advisory с CVE, описанием и credits репортера.
4. Уведомляем пользователей через канал релиза, если затронуты приватные
   данные.

## Что **не** считается уязвимостью

Чтобы экономить время и вам и нам, заранее: следующие классы отчётов
закрываются без вознаграждения.

- Reports от автосканеров без подтверждённого PoC.
- Issue без security-impact: «отсутствует SPF / DMARC на не-mail домене»,
  «можно сделать DoS перегрузкой» (это ожидаемо для public endpoint),
  «нет `Strict-Transport-Security` на staging».
- Социальная инженерия в адрес сотрудников.
- Атаки, требующие предварительной компрометации Telegram-аккаунта
  жертвы.
- Найденные уязвимости в зависимостях, для которых не предложено
  PoC-эксплуатация в нашем продукте (это уже отслеживается через
  Dependabot, см. [`audit-report.md`](docs/security/audit-report.md)).
- DoS-тесты против production без предварительного согласования.

## Bug Bounty

На момент `v1.0.0` денежное вознаграждение **не предусмотрено**, но мы:

- Публично кредитуем репортера в GitHub Security Advisory.
- Добавляем имя в `docs/security/HALL_OF_FAME.md` (будет создан при первом
  репорте).
- Готовы выдать рекомендательное письмо или LinkedIn-связь.
