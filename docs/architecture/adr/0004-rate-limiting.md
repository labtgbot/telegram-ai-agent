# ADR-0004: Rate limiting — Redis sliding window log

- **Статус**: Accepted
- **Дата**: 2026-05-15
- **Авторы**: @konard
- **Связанные документы**: [issue #3](https://github.com/labtgbot/telegram-ai-agent/issues/3), [SECURITY.md](../../SECURITY.md), [C4 Component](../diagrams/c4-component.md)

## Контекст

Лимиты на пользователя:

| Аудитория | В час  | В сутки |
|-----------|--------|---------|
| Free user | 10 запросов | 5 image, 2 video |
| Premium   | 100 запросов| 50 image, 20 video |
| Anonymous | 5 запросов  | — |

Дополнительно: жёсткий лимит на админский логин (5 попыток / 15 минут / IP). Хранилище общее — Redis (уже используется для кэша и брокера Celery).

Алгоритмически рассматриваем три варианта.

## Рассмотренные варианты

### A. Fixed window
Счётчик на ключ `rl:{user}:{HHMM}`, TTL 1 минута.
- Плюсы: проще всего, одна `INCR` команда.
- Минусы: «прыжки» на границе окна — пользователь может сделать 2× лимит за 2 секунды (последняя секунда одного окна + первая следующего). Неприемлемо для image/video с дорогими токенами.

### B. Sliding window counter (gradient)
Взвешенная сумма двух соседних fixed window. Аппроксимация sliding window.
- Плюсы: минимальный объём данных, точность ≈ 99%.
- Минусы: приблизительная — иногда пропускает или ошибочно блокирует. Для бесплатного тарифа допустимо, для платных — спорно.

### C. Sliding window log (sorted set)
Каждый запрос — элемент в Redis `ZSET` с `score = unix_ms`. Перед проверкой удаляем элементы старше окна, считаем оставшиеся.
- Плюсы: точное окно, простая отладка (можно посмотреть лог последних запросов).
- Минусы: O(log N) на операцию + место под N запросов. При лимитах 10/час и 100/час — пренебрежимо.

### D. Token bucket
Бакет с пополнением N токенов/мин, capacity = burst.
- Плюсы: даёт burst-поведение, естественно ограничивает.
- Минусы: для UX «X запросов в час» нужен дополнительный счётчик. Реализация на Redis сложнее (Lua-скрипт с двумя ключами).

## Решение

Принят **Вариант C — sliding window log** для пользовательских лимитов и для админского логина. Реализация:

- Библиотека: [`slowapi`](https://github.com/laurents/slowapi) (FastAPI-обёртка над `limits`) с backend `redis+sliding-window-log`.
- Ключ: `rl:{scope}:{identifier}:{period}` — `scope` ∈ `user|admin|ip`, `period` ∈ `1h|24h|15m`.
- Identifier:
  - Для бота / Mini App: `telegram_id` (после Telegram auth, см. [ADR-0003](./0003-authentication-scheme.md)).
  - Для анонимных эндпоинтов (только health-check): IP.
  - Для админского логина: IP + email (составной ключ, чтобы исключить блокировку всех админов одним IP).
- Sliding window реализуется через `ZADD` + `ZREMRANGEBYSCORE` + `ZCARD` в одном Lua-скрипте (atomic).
- Тариф пользователя читается из `users.is_premium` и кэшируется на 60 секунд.

Структура лимитов:

```python
# core/rate_limits.py
LIMITS = {
    "free":      {"per_hour": 10,  "image_per_day": 5,  "video_per_day": 2},
    "premium":   {"per_hour": 100, "image_per_day": 50, "video_per_day": 20},
    "anonymous": {"per_hour": 5},
    "admin_login": {"per_15m": 5},
}
```

Ответ при превышении: HTTP 429 + заголовок `Retry-After` + JSON `{ "error": "rate_limited", "retry_after": <sec> }`. В Telegram-боте — текст с пояснением и временем сброса.

## Последствия

**Положительные**
- Точные лимиты — нет «прыжков» на границе окна.
- Один и тот же механизм работает для REST, бота и админского логина.
- Лог последних запросов помогает дебажить инциденты («когда именно пользователь упёрся в лимит»).

**Отрицательные / компромиссы**
- Redis становится критическим путём для каждого запроса. Митигируем pipelining, локальным кэшем тарифа, и graceful degradation: при недоступности Redis — `fail open` для пользователей, `fail closed` для админского логина (явный security trade-off).
- Хранение лога O(N) — но N мал (≤ 100 в час на пользователя).

**Out of scope**
- DDoS-защита на L7 — на уровне Cloudflare / nginx (см. [deployment.md](../diagrams/deployment.md)).
- Дифференциация лимитов по моделям AI (Gemini Flash vs Pro) — будет в Phase 2 при настройке pricing.

## Метрики успеха

- 0% ложных 429 при пользовательской нагрузке ≤ лимита.
- < 1 мс p95 на проверку лимита.
- 100% запросов на админский логин с подозрительной частотой блокируются.
- Алёрт в Grafana при росте 429 > 5× от baseline (вероятная атака или баг в Mini App).
