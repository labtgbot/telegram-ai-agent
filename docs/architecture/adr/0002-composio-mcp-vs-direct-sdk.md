# ADR-0002: Composio MCP как единый шлюз к LLM, без прямых SDK провайдеров

- **Статус**: Accepted
- **Дата**: 2026-05-15
- **Авторы**: @konard
- **Связанные документы**: [issue #3](https://github.com/labtgbot/telegram-ai-agent/issues/3), [Composio MCP docs](https://docs.composio.dev/mcp), [C4 Context](../diagrams/c4-context.md)

## Контекст

Продукту нужно работать с несколькими LLM-провайдерами (Gemini, Claude, GPT) и инструментами (search, image, video, voice, document). Возможны два пути:

- **A**: интегрировать каждый провайдер напрямую через его SDK (`google-genai`, `anthropic`, `openai`, `elevenlabs`, плюс отдельные клиенты для web search, image, video).
- **B**: использовать [Composio MCP](https://docs.composio.dev/mcp) как единый шлюз. Composio предоставляет MCP-серверы и унифицированный API к множеству провайдеров и инструментов.
- **C**: гибрид — Composio для редких инструментов, прямые SDK для основных LLM.

Ключевые требования: возможность быстро менять провайдера без релиза, единая точка наблюдаемости запросов, минимальное количество кода для подключения нового инструмента.

## Рассмотренные варианты

### Вариант A — прямые SDK
- Плюсы:
  - Меньше зависимостей от стороннего шлюза.
  - Прямой доступ ко всем фичам каждого провайдера.
  - Нет рисков SLA Composio.
- Минусы:
  - 5–7 разных SDK с разными моделями ошибок, retry-стратегиями, форматами стриминга.
  - Каждое подключение нового инструмента — отдельная интеграция.
  - Биллинг считается в N разных дашбордах провайдеров.
  - Адаптеры/абстракции придётся писать самим — это де-факто пере-изобретение Composio.

### Вариант B — Composio MCP
- Плюсы:
  - Единый интерфейс ко всем провайдерам и инструментам (MCP).
  - Перенос провайдера = конфигурация на стороне Composio, без релиза backend.
  - Один счёт, один dashboard для биллинга AI.
  - Унифицированный формат tool-calls и streaming.
  - Готовые инструменты (web search, document parsing, voice) — не нужно интегрировать каждый отдельно.
- Минусы:
  - Зависимость от SLA Composio и его лимитов.
  - Дополнительный hop по сети (+50–150 мс латентности).
  - Чувствительные данные проходят через сторонний сервис — нужен договор / on-prem.

### Вариант C — гибрид
- Плюсы: гибкость.
- Минусы: усложнение архитектуры без явной выгоды на старте. Требует поддерживать два разных стека.

## Решение

Принят **Вариант B**: единая интеграция с Composio MCP для всех AI-инструментов в Phase 1–3.

**Архитектура**:

- В backend существует один `AIService` (см. [C4 Component](../diagrams/c4-component.md)).
- `AIService` общается **только** с Composio MCP по HTTPS.
- Маппинг «команда пользователя → MCP tool» хранится в конфигурации (`admin_settings.ai_routing`) и обновляется без релиза.
- Каждая вызывающая функция логируется в `token_usage_logs` с полями `composio_tool` и `mcp_server` (уже есть в схеме БД).

## Последствия

**Положительные**
- Один adapter вместо 5–7. Минус ≈70% кода интеграционного слоя.
- Подключение нового инструмента = строка в конфигурации.
- Прозрачный аудит вызовов в единой таблице.
- A/B тестирование моделей делается в Composio dashboard.

**Отрицательные / компромиссы**
- Vendor lock-in на Composio. Митигация: `AIService` спроектирован как порт; теоретическая замена шлюза не затрагивает остальной код.
- Дополнительная сетевая зависимость. Митигация: circuit breaker, таймауты, явные fallback-сообщения пользователю.
- PII в запросах уходит на сторонний сервис. Митигация: подписать DPA с Composio; обсудить on-prem Composio для enterprise-клиентов (Phase 4).

**Out of scope**
- Локальный fallback на open-source LLM (рассмотрим, если SLA Composio станет проблемой).

## Метрики успеха

- 95% AI-запросов завершаются за ≤ 5 сек (image — отдельный лимит).
- Менее 0.5% ошибок интеграции в сутки.
- Подключение нового инструмента занимает < 1 человеко-дня.
- Latency overhead Composio vs прямого SDK ≤ 200 мс p95 (замеряем benchmark'ом каждый квартал).
