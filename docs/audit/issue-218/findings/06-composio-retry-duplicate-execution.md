## Кратко

HTTP-клиент Composio ретраит POST `/tools/execute` в том числе по таймауту
чтения. Для не-идемпотентных тулкитов (например, сабмит видео-генерации) запрос
мог уже выполниться на стороне провайдера, а клиент, не получив ответ вовремя,
повторяет его — провайдер выполняет операцию повторно (двойной сабмит/двойная
оплата провайдеру).

## Доказательства

- `backend/app/services/composio/client.py:169-213` — общий retry-цикл для POST
  без различения идемпотентных/не-идемпотентных операций; ретрай по timeout.
- `backend/app/core/config.py:214-220` — настройки retries/timeout применяются ко
  всем вызовам.
- `backend/app/services/video_generation.py:581-599` — сабмит видео идёт через
  тот же execute-путь, то есть подвержен повторной отправке.

## Влияние

Повторные провайдер-side выполнения: дублированные джобы, лишние списания у
провайдера, рассинхрон между нашим состоянием и состоянием провайдера. Особенно
вреден для платных генераций.

## Предлагаемое исправление

- Не ретраить не-идемпотентные операции по таймауту; либо передавать
  idempotency-key провайдеру, если поддерживается.
- Разделить «безопасные для ретрая» (status/poll) и «небезопасные» (submit)
  вызовы; ретраить только первые.
- Логировать и метрить случаи timeout-без-ответа для submit.

## Критерии приёмки

- [ ] Submit-операции не повторяются автоматически по timeout (или защищены
      idempotency-key).
- [ ] Poll/status по-прежнему ретраятся.
- [ ] Тест проверяет, что timeout на submit не приводит к повторному вызову
      provider execute.
