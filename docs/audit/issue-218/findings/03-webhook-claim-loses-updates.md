# Webhook claim-before-process: сбой commit/dispatch теряет side-effects навсегда и не освобождает Redis-claim

Родительский контекст: #218

| Поле | Значение |
| --- | --- |
| Критичность | High |
| Stage | Stage 1 - High priority |
| Labels | `bug`, `telegram`, `backend`, `stage-1-high`, `complexity-medium` |
| Status | Confirmed |
| GitHub issue | https://github.com/labtgbot/telegram-ai-agent/issues/222 |

## Кратко

Идемпотентность webhook (введённая в #158) «застолбляет» `update_id` в Redis
через `SET NX` **до** обработки апдейта. Если `dispatch_update` или
`session.commit()` падают, код делает rollback БД, логирует и всё равно
возвращает `200 WebhookAck()` — но Redis-claim не снимается. Повторная доставка
того же `update_id` Telegram'ом считается дублем и отбрасывается. Итог: побочные
эффекты (регистрация, начисление реферала и т.п.) теряются безвозвратно.

## Доказательства

- `backend/app/api/v1/bot.py:104-106` — `_claim_update_id` делает `set(..., nx=True)`.
- `backend/app/api/v1/bot.py:128-143` — claim выполняется до обработки; при
  `not claimed` возвращается `WebhookAck()` (дубль отбрасывается).
- `backend/app/api/v1/bot.py:145-162` — `dispatch_update` + `commit`; на любом
  исключении делается `rollback`, но claim в Redis НЕ удаляется, а ответ всё
  равно `200`.
- При redelivery того же `update_id` срабатывает ветка `not claimed` (строки
  141-143) и апдейт молча теряется.

## Влияние

Транзиентный сбой БД/диспетчера на одном апдейте навсегда роняет его эффекты:
потерянные регистрации, не начисленные рефералы, пропущенные команды — без
видимых ошибок (клиенту отдаётся 200, Telegram считает доставку успешной).

## Предлагаемое исправление

- При ошибке обработки освобождать claim (`DEL` ключа) перед возвратом, чтобы
  Telegram повторил доставку; либо возвращать 5xx, чтобы Telegram ретраил.
- Помечать claim как «завершённый» только после успешного commit (claim →
  process → commit → finalize), а до commit держать «in-progress» claim с
  коротким TTL.
- Тест: сбой commit на первом приёме → второй приём того же `update_id`
  обрабатывается и фиксирует эффекты.

## Критерии приёмки

- [ ] Сбой обработки/commit не приводит к молчаливой потере апдейта при
      redelivery.
- [ ] Claim снимается/не финализируется при неуспешной обработке.
- [ ] Идемпотентность сохраняется: успешно обработанный `update_id` не
      обрабатывается дважды.
- [ ] Добавлен регрессионный тест.
