# Видео-джобы без provider_job_id или с постоянной ошибкой провайдера никогда не достигают терминального статуса и не возвращают токены

Родительский контекст: #218

| Поле | Значение |
| --- | --- |
| Критичность | High |
| Stage | Stage 1 - High priority |
| Labels | `bug`, `ai-service`, `composio`, `tokens`, `backend`, `stage-1-high`, `complexity-medium` |
| Status | Confirmed |

## Кратко

Сервис генерации видео списывает токены при сабмите, а затем опрашивает провайдера
в фоне. Если у джобы нет `provider_job_id` (сабмит прошёл, но id не вернулся) или
провайдер при опросе стабильно отдаёт `ComposioError`, джоба навсегда остаётся в
не-терминальном статусе: нет лимита попыток (`attempts`), нет wall-clock timeout и
нет dead-letter перехода. Токены пользователя списаны, видео нет, рефанд не
происходит.

## Доказательства

- `backend/app/services/video_generation.py:418-451` — `_poll_job` на
  `ComposioError` лишь инкрементит `job.attempts`, пишет `error_message`,
  делает `flush()` и `return`; статус джобы не меняется, попытки не ограничены.
- `backend/app/services/video_generation.py:483-486` — ветка `else` оставляет
  джобу в текущем не-терминальном статусе (`queued`/`in_progress`).
- `backend/app/services/video_generation.py:426` — опрос использует
  `job.provider_job_id or ""`: джоба без `provider_job_id` будет опрашиваться с
  пустым id бесконечно.
- `backend/app/workers/video_polling.py:59-76` — `list_active` выбирает только
  не-терминальные джобы; цикл логирует и «глотает» ошибки, поэтому такие джобы
  возвращаются в очередь на каждый проход без верхней границы.
- `backend/app/models/video_job.py` — поле `attempts` есть, но нигде не
  сравнивается с лимитом.

## Влияние

Пользователь теряет токены без результата и без рефанда; «зависшие» джобы
накапливаются и опрашиваются бесконечно, создавая постоянную нагрузку на
провайдера и воркер. Это прямой ущерб токеновой экономике и доверию.

## Предлагаемое исправление

- Ввести `max_attempts` и/или wall-clock deadline (`created_at + N`) в
  `_poll_job`/`list_active`; при превышении — переводить джобу в `failed` и
  вызывать существующий refund-путь (`_apply_failure`).
- Отдельно обрабатывать отсутствие `provider_job_id`: немедленный `failed` +
  refund вместо опроса с пустым id.
- Метрика «stuck video jobs» и алерт.

## Критерии приёмки

- [ ] Джоба с постоянной `ComposioError` при опросе после N попыток/таймаута
      переходит в `failed` и возвращает токены ровно один раз.
- [ ] Джоба без `provider_job_id` завершается терминально с рефандом.
- [ ] Тест воспроизводит «вечную» джобу и проверяет терминализацию + рефанд.
