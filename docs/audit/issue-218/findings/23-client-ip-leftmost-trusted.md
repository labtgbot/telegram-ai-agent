# Client-IP resolver возвращает левый (спуфабельный) XFF-элемент, когда все хопы доверенные

Родительский контекст: #218

| Поле | Значение |
| --- | --- |
| Критичность | Low |
| Stage | Stage 3 - Low priority |
| Labels | `security`, `backend`, `stage-3-low`, `complexity-low` |
| Status | Confirmed |
| GitHub issue | https://github.com/labtgbot/telegram-ai-agent/issues/242 |

## Кратко

`resolve_client_ip` корректно выбирает правый недоверенный хоп. Но если ВСЕ
элементы `X-Forwarded-For` попадают в доверенные сети, функция возвращает
крайний левый элемент (`forwarded_ips[0]`), который контролируется клиентом и
спуфабелен. Это остаточный край после remediation #143.

## Доказательства

- `backend/app/core/client_ip.py:79-81` — корректный путь: правый недоверенный.
- `backend/app/core/client_ip.py:82-83` — fallback: при всех доверенных хопах
  возвращается `forwarded_ips[0]` (левый, спуфабельный).

## Влияние

При широком `TRUSTED_PROXY_IPS` (например, целая подсеть провайдера) клиент за
тем же доверенным сегментом может подделать «client IP», влияя на rate-limit
ключи и audit-IP. Узкий край, отсюда Low.

## Предлагаемое исправление

- В случае «все хопы доверенные» возвращать `peer_host` (фактический peer), а не
  левый XFF-элемент; задокументировать инвариант.

## Критерии приёмки

- [ ] При полностью доверенной цепочке используется реальный peer, не
      клиентский XFF.
- [ ] Тест покрывает all-trusted-цепочку.
